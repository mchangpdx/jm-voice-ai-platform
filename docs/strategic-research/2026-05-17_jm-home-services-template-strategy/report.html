Part 3 — JM Home Services 템플릿화 전략

사용자 제안 요약

"JM Cafe가 FSR/QSR 매장 신규 등록 기준 템플릿으로 쓰였듯, JM Home Services를 home services 업종의 기준 템플릿으로 만들면 비슷한 업종 회사 영입 시 기능/코드 재사용 효과가 큼."

전문가 의견 (Claude Opus 4.7)

사용자 제안에 동의 — 단, 한 단계 더 추상화 권장.

현재 JM Cafe는 FSR/QSR template으로 작동 중. JM Pizza / JM Taco 등록이 거의 0-shot으로 성공한 이유는 다음 5개 자산이 잘 정의되어 있기 때문:

  1. allergen_rules.yaml — vertical 특화 위험 매트릭스
  2. menu.yaml — 서비스 카테고리
  3. modifier_groups.yaml — 옵션 변형
  4. system_prompt_base.txt — 페르소나 + 룰
  5. PHASE FLOW gate (CART → TOTAL → NAME → EMAIL → RECITAL)

JM Home Services는 위 구조의 home services 변형이 아직 미완. 단순히 "JM Home Services를 template화"하는 것보다 "Vertical Template Framework 표준화"를 권장.

모든 vertical (restaurant / home / beauty / auto)이 동일한 9-layer 표준을 따르도록:

LayerJM Cafe (FSR)JM Home Services
1. risk_rules.yamlallergen_rules.yamlsafety_hazard_rules.yaml (가스누출/감전/누수)
2. service_categories.yamlmenu.yamlHVAC/plumbing/electrical 서비스 항목
3. option_groups.yamlmodifier_groups.yamlservice_options.yaml (긴급도/약속시간/추가 진단)
4. system_prompt_base.txtAria/Sofia/Marco 페르소나Rex 페르소나 + dispatch 룰
5. intake_flow.yaml (NEW)CART → TOTAL → NAME → EMAIL → RECITALINTAKE → DIAGNOSIS → ESTIMATE → SCHEDULE → CONFIRM

8. crm_followup.yaml (NEW)"the usual" 제안year-over-year 정기 점검 reminder

왜 이렇게 추상화하면 더 좋은가?

  • 모든 vertical이 같은 9 자산만 작성하면 등록 가능 → 새 vertical 추가 1-2일 (현재는 1-2주)
  • 코드 재사용률 80%+ (UI/통화/DB/결제 공유)
  • JM Home Services 등록 = home services 전체 vertical의 reference implementation
  • 다음 ABC Plumbing 등록 → vertical=home_services 선택만으로 80% 작동
  • HVAC 회사 추가 → system_prompt에 1-2줄만 수정
  • 영업 비용 시간 대폭 감소 → Avoca와 직접 경쟁 가능

더 좋은 방법 (가감없이 솔직히):

  1. Layer 5 intake_flow.yaml가 가장 중요 — 이미 PHASE FLOW gate가 system prompt에 있지만 vertical 별로 단계가 다름. 이를 vertical config로 추상화하면 LLM이 vertical 따라 자동으로 흐름 변경.
  2. Layer 6 scheduler.yaml — 가장 큰 갭. 음식점은 table reservation 단순 시간 슬롯이지만 home services는 technician × geo × skill match. ServiceTitan이 잘하는 영역.
  3. JM Home Services는 reference + 실제 회사 자료로 가치 검증 필요. JM Cafe는 실제 카페이고 통화 검증됨. JM Home Services는 데모 매장으로 phone도 routing 미연결 (DB 확인 결과). 실제 영업 1-2 가맹점 확보 후 라이브 검증해야 template 신뢰도 ↑.
  4. Avoca의 코드/방식을 따라하지 말 것. Avoca는 ServiceTitan dependency. JM은 자체 backbone — 단점은 dispatch 약점, 장점은 multi-vertical + SMB 진입 가능. "Avoca보다 작은 매장, 그러나 multi-vertical, 한국어 + 영어" 포지션이 가장 강함.
  5. 1차 출시는 home services 보류 권장. FSR/QSR (Loyverse + SumUp) 출시로 매출 확보 → home services는 2-3차에 본격. 자원 분산 위험.

Part 7 — Multi-vertical Architecture Proof 전략

사용자 의도 요약

"멀티-버티컬 설계가 실제로 다른 분야에 적용했을 때 얼마큼 효과적인지 검증하고 싶다. FSR/QSR에서는 이 설계가 큰 힘을 발휘하는 게 데이터상 검증됐다. 하지만 전혀 다른 분야에 시도 안 해봐 확신이 없다. Home services를 MVP로 빠르게 검증하려 했다."

전문가 의견 (가감없이) — Home services는 MVP 검증 vehicle로 부적합

사용자 의도(다른 분야 검증)는 100% 동의. FSR/QSR 안에서 cafe → pizza → mexican 매장 추가된 건 같은 "음식 주문" 패턴 변주 (메뉴 + 알러지 + modifier). 전혀 다른 산업 패턴 (서비스 예약 + 견적 + 시간 슬롯) 검증이 진짜 architecture proof이고, 이게 영업·투자자 pitch의 가장 강력한 무기.

그러나 Home services를 MVP로 선택한 판단은 다시 검토 권장. 다음 5개 검증 우려 요소 때문:

우려 요소영향
Dispatch/Technician scheduling 복잡MVP에 진짜 minimum 작업이 어려움 — HS1-HS4 P0 4건 + 신규 tool 3개 + 9-layer yaml 필요 = 6-8주. MVP라기엔 무겁다.
기존 인프라 의존성Home services SMB의 60-70%가 이미 ServiceTitan / HCP / Workiz / Jobber 사용 중. 우리는 이들의 native integration이 없어 영업 진입 장벽 ↑.
Sales cycle 길음Home services 사업자는 보수적 buyer (한 번 시스템 바꾸면 1년+ 사용). "빠른 검증"을 위한 가맹점 1개 확보까지 영업 사이클 1-3개월. MVP 빠른 iteration 불가.
자원 분산 위험1차 FSR/QSR 출시 4-5주 작업 한가운데에서 새 vertical 작업 = 안정성 ↓. SumUp PG / staging environment 등 P0 작업 지연 위험.
데이터 다양성 부족1-2개 home services 매장만으로는 "architecture 검증" 효과 약함. Avoca/투자자에게 "1 가맹점에서 작동"은 proof 안 됨.

솔직한 판단: Home services는 장기 핵심 시장 (Avoca $1B 증거)이지만 MVP 검증 vehicle로는 비효율. 차라리 가벼운 vertical로 1-2주 안에 빠르게 architecture proof → 그 결과를 영업/투자자에게 보여주고 → Home services에 본격 진입하는 게 효율 최적.

Multi-vertical Architecture Proof의 진짜 가치

  • 영업 자료: "우리는 음식점뿐 아니라 미용실/자동차/홈서비스 모두 동일 코드베이스로 서비스" → 1매장 영입 시 다른 vertical 함께 영업 가능
  • 투자자 pitch: "9-layer Vertical Template = horizontal SaaS 구조" → SaaS valuation 적용 가능 (vertical-only는 평가 절상폭 작음)
  • 기술 부채 감소: Framework 표준화하면 각 vertical 추가 시 코드 변경 없이 yaml만 작성 — 유지비 ↓
  • 경쟁 차별점: Avoca/Maple은 vertical-specific. JM은 horizontal — 같은 Korea-friendly multi-vertical 경쟁자 없음

Beauty Salon 추천 이유 (8가지)

  1. 음식점 패턴과 가장 다름 — order(stack of items + payment) vs appointment(time slot + service + stylist). Architecture 검증 가치 최대. PHASE FLOW도 완전 다른 흐름 (메뉴 주문 → 진단 + 슬롯 선택).
  2. 구현 난이도 가장 낮음 — dispatch 복잡 없음, technician skill match 없음 (stylist 1-2명 = "any available"), emergency triage 없음. 음식점 코드 80% 재사용 + tool 1-2개 추가만 신규.
  3. SMB 비율 매우 높음 — 미국 미용실의 90%+가 1-5인 매장. Avoca/대형 솔루션이 진입 불가. JM의 "SMB-friendly + 5lang" 강점 직접 발휘.
  4. Phone-first 산업 — 미용실 80%+가 예약을 phone로 받음. AI voice agent 핵심 가치 (24/7, 영어 못하는 customer 응대) 즉시 발현.
  5. 가맹점 수 매우 많음 — 미국 100,000+ 매장 (BLS 통계). 영업 잠재력 풍부 — 한 area에서 10-20 매장 영입 가능.
  6. MVP 가능 (1-2주) — Beauty template 9-layer 작성 + 1-2 tool 신규 + JM Beauty Salon 라이브 검증 = 10-14일. Home services 6-8주 대비 5분의 1.
  7. 다국어 효과 가장 큼 — 한국계 미용실 (LA/NY/Atlanta/Texas) 수천 개, 라틴계 미용실 (Spanish) 수만 개, 베트남계 네일살롱 수만 개. JM의 5lang이 가장 큰 무기가 되는 vertical.
  8. 이미 DB에 JM Beauty Salon 매장 등록됨 — system_prompt = Luna 페르소나 + agency_id + business_hours 모두 채워져 있음. Wizard 자료 + DB 모두 준비된 상태에서 출발.
